<objective>
## Goal
Create the `useCashBook` composable and the Kassenbuch page with: Barkassen management (create, select, assign machines), KPI cards (current balance, total withdrawals, total corrections, integrity check), entries table with time filter, and theoretical cash display. This plan builds the page shell — entry dialogs and PDF export follow in Plan 02.

## Purpose
Operators need a central place to view their Barkassen, see the current cash balance, browse the GoBD-compliant entry history, and assign machines to Barkassen.

## Output
- `useCashBook.ts` composable with full CRUD + theoretical cash
- `/cash-book` page with Barkassen selector, KPI cards, entries table
- Sidebar navigation entry
- i18n translations (de/en)
</objective>

## AC-1: useCashBook composable
```gherkin
Given the composable is imported
When fetchCashBooks() is called
Then it loads all Barkassen for the company with their latest balance
And provides createCashBook(name, initialBalance) to create a new Barkasse
And provides fetchEntries(cashBookId, dateRange?) to load entries
And provides getTheoreticalCash(cashBookId) calling the RPC
And provides assignMachine(machineId, cashBookId) and unassignMachine(machineId)
```

## AC-2: Kassenbuch page with Barkassen selector
```gherkin
Given the user navigates to /cash-book
When the page loads
Then it shows a dropdown/selector to pick a Barkasse (or create a new one)
And the selector shows Barkasse name + current balance
And if only one Barkasse exists, it's auto-selected
And a "Neue Barkasse" button opens a creation dialog
```

## AC-3: KPI cards
```gherkin
Given a Barkasse is selected
When entries are loaded
Then 4 KPI cards are shown:
  - "Aktueller Stand" with current balance (last entry's balance_after)
  - "Gesamtentnahmen" with sum of withdrawal amounts + count
  - "Gesamtkorrekturen" with sum of correction amounts + count
  - "Integritätsprüfung" showing verified/total entries count
```

## AC-4: Entries table with filters
```gherkin
Given a Barkasse is selected
When entries are loaded
Then a table shows: Datum, Art (badge), Betrag (colored), Stand danach, Beschreibung, Ausgeführt von
And reversed entries show a warning row with "Differenz" info (if counted_amount differs from expected)
And a date range filter (Letzte 30 Tage, Letzte 90 Tage, Dieses Jahr, Alle) controls the query
And entries are sorted newest-first
```

  <name>Task 1: Create useCashBook composable</name>
  <files>management-frontend/app/composables/useCashBook.ts</files>
  <action>
    Create composable following the useMachineTrays pattern:

    **Interfaces (at top of file):**
    ```
    CashBook: { id, name, company_id, initial_balance, activated_at, created_by, is_active, created_at }
    CashBookEntry: { id, created_at, cash_book_id, company_id, entry_number, type, amount, balance_after, description, machine_id, counted_amount, expected_amount, corrects_entry_id, is_reversed, created_by, hash }
    TheoreticalCash: { theoretical_balance, last_entry_balance, cash_sales_since, last_entry_at, entry_count, machines: { machine_id, machine_name, cash_sales }[] }
    ```

    **Reactive state:**
    - cashBooks: ref<CashBook[]>([])
    - selectedCashBook: ref<CashBook | null>(null)
    - entries: ref<CashBookEntry[]>([])
    - theoreticalCash: ref<TheoreticalCash | null>(null)
    - loading: ref(false)
    - entriesLoading: ref(false)

    **Functions:**
    - fetchCashBooks(): Supabase select from cash_books, order by name
    - createCashBook(name: string, initialBalance: number): Insert into cash_books with company_id and created_by from auth. After insert, refetch.
    - fetchEntries(cashBookId: string, options?: { from?: string, to?: string }): Select from cash_book_entries where cash_book_id, optional date filter, order by entry_number DESC. Join auth.users to get created_by display name — BUT since we can't join auth.users via PostgREST, store created_by as UUID and resolve names client-side using a simple cache map (fetch from organization_members).
    - fetchTheoreticalCash(cashBookId: string): Call RPC get_theoretical_cash with cash_book_id and company_id
    - createEntry(entry: { cash_book_id, type, amount, description?, machine_id?, counted_amount?, expected_amount?, corrects_entry_id? }): Insert into cash_book_entries with company_id and created_by. After insert, refetch entries + theoretical cash.
    - assignMachine(machineId: string, cashBookId: string): Update vendingMachine set cash_book_id where id = machineId
    - unassignMachine(machineId: string): Update vendingMachine set cash_book_id = null where id = machineId
    - fetchAssignedMachines(cashBookId: string): Select from vendingMachine where cash_book_id
    - fetchAllMachines(): Select all vendingMachine for company (for assignment dialog)
    - verifyIntegrity(entries: CashBookEntry[]): Client-side hash chain verification. Recompute SHA256 for each entry and compare against stored hash. Return { verified: number, total: number, valid: boolean }. NOTE: Use Web Crypto API (crypto.subtle.digest) since this runs in browser — do NOT import node:crypto or pgcrypto.

    **User name resolution:**
    - memberNames: ref<Record<string, string>>({})
    - fetchMemberNames(): Select from organization_members joined with users table (or use Supabase auth.users admin API if accessible). Cache by user_id. If auth.users not accessible, just show "User" as fallback — plan 02 can refine.
    - getMemberName(userId: string): return memberNames[userId] or 'Unbekannt'

    **Activity logging:**
    - Log cash book creation and entry creation to history table (same pattern as useMachineTrays logActivity)

    Avoid:
    - Do NOT import from 'crypto' or 'node:crypto' — use Web Crypto API (crypto.subtle) for browser SHA256
    - Do NOT use Supabase realtime subscriptions (entries are append-only, no need for live updates)
    - Do NOT add update/delete functions for entries (GoBD immutability)
    - Cast Supabase query results manually (no generated types, same pattern as other composables)
  </action>
