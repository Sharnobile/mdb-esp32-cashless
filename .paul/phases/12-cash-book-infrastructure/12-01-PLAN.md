<objective>
## Goal
Create the database schema for a GoBD-compliant Barkassen (cash register) system. A company has 1-n Barkassen, each Barkasse has 0-n vending machines assigned. Entries are immutable with a SHA256 hash chain. Theoretical cash is calculated across all machines of a Barkasse.

## Purpose
Operators need a cash register system to track cash flow from vending machines. Small operators start with one Barkasse for all machines; larger operators can have regional Barkassen (e.g. "Berlin", "München"). The system must be GoBD-compliant (immutable, verifiable, sequential).

## Output
- Migration file `20260407000000_cash_book.sql`
- `cash_books` table (Barkassen per company, with name)
- `cash_book_entries` table (immutable entries with hash chain)
- `cash_book_id` column on `vendingMachine` (machine → Barkasse assignment)
- `get_theoretical_cash(cash_book_id)` RPC
- Trigger functions for hash chain, auto-numbering, reversal handling
</objective>

<acceptance_criteria>

## AC-1: Barkasse creation with 1-n per company
```gherkin
Given a company
When an admin creates a Barkasse with a name and initial balance
Then a cash_books row is created with name, initial_balance, activated_at
And an automatic "initial" entry is created in cash_book_entries
And the company can have multiple Barkassen (no UNIQUE on company_id)
```

## AC-2: Machine assignment to Barkasse
```gherkin
Given a Barkasse and vending machines
When machines are assigned to the Barkasse
Then vendingMachine.cash_book_id is set to the Barkasse's id
And a machine can only be assigned to one Barkasse at a time
And unassigned machines (cash_book_id=NULL) are allowed
```

## AC-3: Immutable entries with hash chain
```gherkin
Given an existing Barkasse with entries
When a new entry is inserted (withdrawal, correction, payout, reversal)
Then entry_number is auto-incremented (sequential, no gaps per Barkasse)
And balance_after is auto-calculated (previous balance + amount)
And hash is auto-calculated as SHA256(entry_number || type || amount || balance_after || previous_hash)
And UPDATE and DELETE are denied on cash_book_entries via RLS
```

## AC-4: GoBD-compliant reversal (Stornobuchung)
```gherkin
Given an existing entry with a mistake (e.g. typo)
When the user creates a reversal
Then a new entry of type 'reversal' is created with:
  - amount = negated original amount
  - corrects_entry_id = id of the wrong entry
And the original entry is NOT modified or deleted
And the original entry's is_reversed flag is set to true by trigger
And reversing an already-reversed entry is rejected
```

## AC-5: Theoretical cash calculation across all assigned machines
```gherkin
Given a Barkasse with 3 machines assigned and cash sales on each
When get_theoretical_cash(cash_book_id, company_id) is called
Then it returns:
  - last_entry_balance: balance_after of the most recent entry
  - cash_sales_since: SUM of cash sales (channel='cash') across ALL assigned machines since the last entry
  - theoretical_balance: last_entry_balance + cash_sales_since
  - per_machine breakdown: array of { machine_id, machine_name, cash_sales } for each assigned machine
```

## AC-6: Multi-tenancy
```gherkin
Given two companies each with Barkassen
When company A queries cash_books or cash_book_entries
Then only company A's data is returned (RLS via my_company_id())
```

  <action>
    Create migration file with:

    **A. cash_books table (= Barkassen):**
    - id (uuid PK, gen_random_uuid)
    - created_at (timestamptz, default now)
    - company_id (uuid FK → companies, ON DELETE CASCADE, NOT NULL)
    - name (text NOT NULL) — e.g. "Barkasse Berlin", "Hauptkasse"
    - initial_balance (float8 NOT NULL DEFAULT 0) — Anfangsbestand / starting change money
    - activated_at (timestamptz NOT NULL DEFAULT now) — GoBD activation timestamp
    - created_by (uuid FK → auth.users, NOT NULL)
    - is_active (boolean NOT NULL DEFAULT true)
    - NO unique constraint on company_id (1-n Barkassen per company)
    - UNIQUE (company_id, name) — names must be unique within a company

    RLS policies:
    - SELECT: authenticated, company_id = my_company_id()
    - INSERT: authenticated, company_id = my_company_id()
    - UPDATE: authenticated, company_id = my_company_id() — only name + is_active can change
    - DELETE: NONE (GoBD: Barkassen cannot be deleted)

    **B. cash_book_entries table:**
    - id (uuid PK, gen_random_uuid)
    - created_at (timestamptz NOT NULL DEFAULT now)
    - cash_book_id (uuid FK → cash_books, ON DELETE RESTRICT, NOT NULL)
    - company_id (uuid NOT NULL FK → companies) — denormalized for RLS
    - entry_number (integer NOT NULL) — auto-populated by trigger, sequential per cash_book_id
    - type (text NOT NULL) — one of: 'initial', 'withdrawal', 'correction', 'payout', 'reversal'
    - amount (float8 NOT NULL) — positive for additions, negative for withdrawals/payouts
    - balance_after (float8 NOT NULL) — auto-populated by trigger
    - description (text) — free text, e.g. "Geldentnahme - Bankeinzahlung"
    - machine_id (uuid, nullable, FK → "vendingMachine") — which machine was cash withdrawn from (optional, for withdrawals)
    - counted_amount (float8) — nullable, what was actually counted (Soll/Ist comparison)
    - expected_amount (float8) — nullable, theoretical amount snapshot at time of entry
    - corrects_entry_id (uuid, nullable, FK → cash_book_entries self-reference) — set when type='reversal'
    - is_reversed (boolean NOT NULL DEFAULT false) — set by trigger when a reversal references this entry
    - created_by (uuid FK → auth.users, NOT NULL)
    - hash (text NOT NULL) — SHA256 hex, auto-populated by trigger
    - UNIQUE (cash_book_id, entry_number)
    - CHECK: (type = 'reversal' AND corrects_entry_id IS NOT NULL) OR (type != 'reversal' AND corrects_entry_id IS NULL)

    RLS policies:
    - SELECT: authenticated, company_id = my_company_id()
    - INSERT: authenticated, company_id = my_company_id()
    - UPDATE: NONE (GoBD: entries are immutable)
    - DELETE: NONE (GoBD: entries cannot be deleted)

    Grant service_role select, insert on both tables (no update/delete on entries).
    Grant authenticated select, insert on both tables; select, update on cash_books only.

    **C. Add cash_book_id to vendingMachine:**
    - ALTER TABLE "vendingMachine" ADD COLUMN cash_book_id uuid REFERENCES cash_books(id) ON DELETE SET NULL;
    - Nullable — machines don't have to be assigned
    - Index: CREATE INDEX idx_vending_machine_cash_book ON "vendingMachine"(cash_book_id);
    - No RLS change needed (existing vendingMachine RLS already handles company scope)

    **D. Trigger function: before_insert_cash_book_entry()**
    - Lock the cash_book row (SELECT FOR UPDATE) to prevent concurrent inserts
    - Calculate entry_number: MAX(entry_number) + 1 for this cash_book_id (or 1 if first)
    - Calculate balance_after: previous balance_after + NEW.amount (or NEW.amount if first entry)
    - Get previous hash (or empty string '' if first entry)
    - Calculate hash: encode(digest(entry_number::text || type || amount::text || balance_after::text || previous_hash, 'sha256'), 'hex')
    - Set NEW.entry_number, NEW.balance_after, NEW.hash
    - **Reversal handling**: If NEW.type = 'reversal' AND NEW.corrects_entry_id IS NOT NULL:
      - Validate that corrects_entry_id belongs to the same cash_book_id
      - Validate that the referenced entry is not already reversed (is_reversed = false) — RAISE EXCEPTION if so
      - UPDATE the referenced entry: SET is_reversed = true (trigger runs as SECURITY DEFINER, bypasses RLS)
      - Auto-set NEW.amount = negated amount of referenced entry (override any client-provided amount)
    - Requires pgcrypto extension (CREATE EXTENSION IF NOT EXISTS pgcrypto)

    **E. Auto-create initial entry trigger: after_insert_cash_book()**
    - On INSERT into cash_books, auto-insert an entry with:
      - type='initial', amount=NEW.initial_balance, description='Anfangsbestand'
      - company_id=NEW.company_id, cash_book_id=NEW.id, created_by=NEW.created_by
    - This ensures every Barkasse starts with an "Anfangsbestand" entry

    Avoid:
    - Do NOT use SERIAL for entry_number (trigger-based to guarantee no gaps)
    - Do NOT add UPDATE/DELETE policies on cash_book_entries (GoBD immutability)
    - Do NOT use CASCADE on cash_book_entries FK to cash_books (use RESTRICT)
    - Do NOT allow reversing an already-reversed entry (prevent double-Storno)
    - Do NOT make cash_book_id on vendingMachine NOT NULL (backward compat: existing machines have no Barkasse)

    **RPC: get_theoretical_cash(p_cash_book_id uuid, p_company_id uuid)**
    - Returns JSON: { theoretical_balance, last_entry_balance, cash_sales_since, last_entry_at, entry_count, machines: [{machine_id, machine_name, cash_sales}] }
    - SECURITY DEFINER
    - Manual p_company_id check against auth.uid() → organization_members
    - Logic:
      1. Get the latest cash_book_entry for this cash_book:
         SELECT balance_after, created_at FROM cash_book_entries
         WHERE cash_book_id = p_cash_book_id
         ORDER BY entry_number DESC LIMIT 1
      2. Get all machines assigned to this Barkasse:
         SELECT vm.id, vm.name FROM "vendingMachine" vm
         WHERE vm.cash_book_id = p_cash_book_id
      3. For each machine, sum cash sales since last entry:
         SELECT COALESCE(SUM(item_price), 0) FROM sales
         WHERE machine_id = vm.id AND channel = 'cash'
         AND created_at > last_entry_at
      4. cash_sales_since = total across all machines
      5. theoretical_balance = last_entry_balance + cash_sales_since
      6. Return as JSON with per-machine breakdown

    If no cash book exists, return NULL.
    If no entries exist (should not happen due to auto-initial), return NULL.

    Avoid:
    - Do NOT use RLS-aware client (SECURITY DEFINER bypasses RLS)
    - Do NOT forget the company_id membership check
