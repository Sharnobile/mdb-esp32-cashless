## Goal
Build the Stripe payment backend infrastructure: DB schema for per-company Stripe keys and payment records, edge functions for PaymentIntent creation + payment confirmation + webhook handling, and settings UI for operators to configure their Stripe credentials.

## Purpose
Enables vending machine operators to accept mobile payments through their own Stripe account. Foundation for Phase 15 Plan 02 (payment UI on public page).

## Output
- DB migration: Stripe columns on companies + payments table
- 3 edge functions: create-payment-intent, confirm-payment, stripe-webhook
- Settings page: Stripe key management UI

## Architecture (from DISCOVERY.md)
- Stripe `npm:stripe@^13.0.0` for Deno edge functions
- PaymentIntents API with per-company secret keys
- Hybrid confirmation: client-side primary + webhook backup
- Credit delivery via mqttPublish() shared module (same XOR encryption as send-credit)
- Idempotent payment processing using PaymentIntent ID as dedup key
- Webhook URL: `/functions/v1/stripe-webhook?company_id={uuid}` for multi-tenant signature verification

## AC-3: Payment confirmation with credit delivery
```gherkin
Given a successful Stripe payment for product "Coca-Cola" on machine "Büro 2. OG"
When POST /functions/v1/confirm-payment is called with { payment_intent_id, subdomain }
```

## AC-4: Webhook handles missed payments
```gherkin
Given a payment_intent.succeeded webhook event from Stripe
When POST /functions/v1/stripe-webhook?company_id={uuid} is called
Then the webhook signature is verified using the company's stripe_webhook_secret
And if the payment is not already recorded (idempotent check), credit is delivered
And a payment record is created
```

## AC-5: Duplicate payment prevention
```gherkin
Given a payment that was already confirmed (record exists in payments table)
When the same PaymentIntent is confirmed again (via confirm-payment or webhook)
Then no duplicate credit is delivered
And the response indicates already processed
```

  <action>
    **All deno.json files**: Use npm import for Stripe:
    ```json
    { "imports": { "@supabase/supabase-js": "https://esm.sh/@supabase/supabase-js@2", "stripe": "npm:stripe@^17" } }
    ```

    **create-payment-intent/index.ts** (public, no auth):
    - Accept POST with JSON: `{ subdomain, product_id, slot }`
    - Use service_role Supabase client to:
      1. Find embedded by subdomain → get company
      2. Find machine by embedded
      3. Find tray by machine_id + item_number=slot → get product + sellprice
      4. Get company's stripe_secret_key and stripe_publishable_key
      5. If no Stripe key configured → return 503 "Payment not available for this machine"
    - Create Stripe instance with company's secret key
    - Create PaymentIntent:
      - amount: Math.round(sellprice * 100) (EUR → cents)
      - currency: 'eur'
      - metadata: { machine_id, product_name, slot, subdomain, company_id }
    - Return: `{ clientSecret, paymentIntentId, publishableKey }`
    - CORS headers

    **confirm-payment/index.ts** (public, no auth):
    - Accept POST with JSON: `{ payment_intent_id, subdomain }`
    - Use service_role Supabase client to:
      1. Check payments table — if record with this PI ID exists AND credit_delivered=true → return { success: true, status: "already_processed" }
      2. Find embedded by subdomain → get company, device_id, passkey
      3. Get company's stripe_secret_key
      4. Create Stripe instance, retrieve PaymentIntent by ID
      5. Verify PI status === 'succeeded'
      6. Verify PI metadata.company_id matches (prevent cross-company abuse)
    - Deliver credit:
      1. Build 19-byte XOR payload (same as send-credit): cmd=0x20, amount from PI, timestamp, checksum
      2. XOR encrypt with device passkey
      3. mqttPublish(`/${company_id}/${device_id}/credit`, payload, { qos: 1 })
    - Record payment:
      - INSERT into payments table with credit_delivered=true, credit_delivered_at=now()
      - Use ON CONFLICT (stripe_payment_intent_id) DO NOTHING for idempotency
    - Return: `{ success: true, status: "credit_delivered" }`
    - CORS headers

    **stripe-webhook/index.ts** (public, no auth — called by Stripe):
    - Accept POST with raw body + Stripe-Signature header
    - Read company_id from query parameter
    - Use service_role Supabase client to get company's stripe_webhook_secret
    - Verify webhook signature using Stripe.webhooks.constructEventAsync()
    - Handle event type 'payment_intent.succeeded':
      1. Check payments table for existing record → if credit_delivered=true, skip (idempotent)
      2. If no record or credit not delivered:
         - Look up machine by PI metadata.machine_id
         - Get device passkey from embeddeds
         - Build + XOR encrypt credit payload
         - Publish via MQTT
         - Upsert payment record with credit_delivered=true
    - Return 200 immediately (Stripe requires fast response)
    - No CORS needed (Stripe server-to-server)

    **Shared credit delivery logic**: Extract the XOR payload building + MQTT publishing into a helper function within confirm-payment or create a small shared module. The logic is:
    ```typescript
    async function deliverCredit(supabase, machineId: string, amountCents: number) {
      // 1. Get machine → embedded → passkey
      // 2. Build 19-byte payload: [0x20, 0x01, amount(4), itemNumber(2), timestamp(4), random(6), checksum]
      // 3. XOR encrypt bytes [1..18] with passkey
      // 4. mqttPublish(`/${company_id}/${device_id}/credit`, payload, { qos: 1 })
    }
    ```
    Reference `send-credit/index.ts` lines 104-130 for exact payload format.
