<objective>
## Goal
Migrate the public machine page URL scheme from device-based `embeddeds.subdomain` (bigserial) to stable machine UUIDs (`vendingMachine.id`), and add a per-machine `public_listing` privacy flag. Fix the architectural flaw where replacing a broken ESP32 device would invalidate all printed QR codes.

## Purpose
Stable URLs that survive device swaps. QR codes printed on physical machines stay valid when the ESP32 inside gets replaced. Plus a privacy flag that lets operators hide specific machines from future discovery views (global map, operator page) — while keeping the direct URL accessible for employees.

## Output
- DB migration: `vendingMachine.public_listing` column
- Updated edge functions: `public-machine-data`, `create-payment-intent`, `confirm-payment` use `machine_id` instead of `subdomain`
- Renamed Nuxt route: `/m/[machine_id]` instead of `/m/[subdomain]`
- All payment + notification flows work with the new URL scheme
</objective>

## Architecture Decision
**Why UUID and not a new short column?**
- No migration of existing data needed (UUID already exists)
- Not enumerable — prevents `/m/1`, `/m/2`, `/m/3`... discovery by competitors
- Globally unique across companies
- QR codes make URL length irrelevant (nobody types UUIDs manually)

**Why now?**
- Phase 14 shipped just today — no operator has printed QR codes yet
- Breaking the URL scheme now is zero-cost
- Waiting one day risks operators starting to distribute the old-format URLs

## Edge Functions Currently Using `subdomain`
- `public-machine-data` — accepts `subdomain` as query param
- `create-payment-intent` — accepts `subdomain` in JSON body, uses it to look up device
- `confirm-payment` — accepts `subdomain` in JSON body, uses it for device lookup + validation
- `stripe-webhook` — NO change needed, already uses `machine_id` from PI metadata

## Edge Functions NOT Using `subdomain`
- `subscribe-restock` — already uses `machine_id` in body
- `submit-product-wish` — already uses `machine_id` in body
- `stripe-webhook` — already uses `machine_id` from PI metadata

## Why public_listing Column?
Operator might have mostly public machines (shopping centers, train stations) plus one private machine (inside a company office). Direct URL still works (employees scan QR), but discovery views (future `/m/` global map, `/m/o/[operator]` page) respect the flag.
