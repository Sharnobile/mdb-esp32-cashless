<objective>
## Goal
Build the per-operator page at `/m/o/[company_id]` showing all public machines of a specific operator, plus add a "back to operator machines" link on the single machine detail page `/m/[machine_id]`.

## Purpose
Customers who found one machine (e.g., via QR code) can discover other machines from the same operator. If they like SnackFlow's service at one location, they can find other SnackFlow machines nearby. Operators get their own "mini-brand page" reachable from every machine.

## Output
- Extended `public-machines-list` with optional `company_id` filter + company object in response
- Extended `public-machine-data` with `company_name` field in response
- New page `/m/o/[company_id].vue` with company-specific map + list
- Back navigation banner on `/m/[machine_id].vue`
- i18n translations
</objective>

    **public-machines-list/index.ts**:

    1. Accept optional query parameter `company_id`
    2. Validate as UUID if provided (use UUID_RE regex)
    3. If `company_id` provided:
       - Filter query: `.eq('company', company_id)`
       - Additionally fetch company name: separate query to `companies` table for `{ id, name }` (needed for empty list case)
       - Return response as `{ company: { name }, machines: [...] }`
    4. If `company_id` NOT provided:
       - Keep existing behavior
       - Return `{ machines: [...] }` (no company field)
    5. If `company_id` provided but company does not exist:
       - Return 404 `{ error: 'Operator not found' }`

    **public-machine-data/index.ts**:

    1. The existing query already fetches `machine.company` and a separate company query for stripe keys
    2. Extend the companies query to also select `name`:
       ```typescript
       const { data: company } = await supabase
         .from('companies')
         .select('name, stripe_publishable_key')
         .eq('id', machine.company)
         .single()
       ```
    3. Add `company_name: company?.name ?? null` to the response object
    4. Keep existing `company_id` field (already added in Plan 01)

    No new UUID validation needed for public-machine-data (already validates machine_id).
  </action>

    2. curl `public-machines-list?company_id=[valid-uuid]` — returns { company: { name }, machines: [...] }
    3. curl `public-machines-list?company_id=invalid` — returns 400
    4. curl `public-machines-list?company_id=[nonexistent-uuid]` — returns 404
    5. curl `public-machine-data?id=[machine-uuid]` — response includes company_name
  </verify>
