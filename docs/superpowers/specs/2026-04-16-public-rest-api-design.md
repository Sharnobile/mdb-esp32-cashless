# Public REST API Design

**Date:** 2026-04-16
**Status:** Draft

## Summary

Add a public REST API (`/api/v1/`) that external systems can use with API keys to access all company data (CRUD) and trigger device actions. Uses a thin API gateway edge function that converts API keys to short-lived JWTs, then proxies to PostgREST where existing RLS policies enforce company isolation. OpenAPI/Swagger documentation served at `/api/v1/docs`.

## Goals

- Expose all company data (except auth/user management) via standard REST API
- Use existing API key infrastructure (`api_keys` table, `vmf_` prefix, SHA-256 hashing)
- Minimize maintenance: new DB tables/columns automatically available via PostgREST
- Security via existing RLS policies — no per-endpoint authorization code
- Interactive API documentation via Swagger UI

## Non-Goals

- API key scopes (read-only vs read-write) — can be added later
- Per-endpoint rate limiting — uniform rate limit per key for now
- Webhooks/callbacks for external systems — separate feature
- GraphQL API — out of scope

## Architecture

```
External Client
    │
    │  X-API-Key: vmf_xxx
    │  GET /api/v1/sales?limit=10
    ▼
┌─────────────┐
│    Kong      │  Route: /api/v1/* → Edge Function "api-v1"
└──────┬──────┘
       ▼
┌──────────────────┐
│  api-v1 Gateway  │  1. Validate API key (SHA-256 lookup in api_keys)
│  (Edge Function) │  2. Check rate limit
│                  │  3. Create short-lived JWT (60s, with company_id claim)
│                  │  4. Route: CRUD → PostgREST, Actions → Edge Functions
└──────┬───────────┘
       │
       ├─── CRUD requests ──→ PostgREST (JWT + RLS = company isolation)
       │
       └─── Action requests ──→ Existing edge functions (via service-role + company context)
```

### Key Principle

The gateway converts `X-API-Key` → JWT. After that, PostgREST + RLS handle everything. No new security code per table.

## Database Changes

### Migration: Extend `my_company_id()` and `i_am_admin()`

New migration file: `YYYYMMDDHHMMSS_api_gateway_auth.sql`

The gateway creates JWTs with custom claims (`api_company_id`, `api_key_id`). The existing RLS helper functions need to recognize these claims as a second authentication path.

**Critical constraint: both functions MUST remain `LANGUAGE sql STABLE`** to preserve PostgreSQL's ability to inline them into RLS policy checks. Switching to PL/pgSQL would cause a performance regression on every RLS-protected query across the entire system.

```sql
-- Extend my_company_id() to support API key JWTs
-- Stays LANGUAGE sql for RLS inlining performance
CREATE OR REPLACE FUNCTION my_company_id() RETURNS uuid LANGUAGE sql STABLE AS $$
  SELECT COALESCE(
    -- 1. Normal user flow (existing behavior)
    (SELECT company_id FROM organization_members WHERE user_id = auth.uid() LIMIT 1),
    -- 2. API key flow: read company_id from JWT claim
    (current_setting('request.jwt.claims', true)::json->>'api_company_id')::uuid
  )
$$;

-- Extend i_am_admin() to grant admin rights to API key holders
-- Stays LANGUAGE sql for RLS inlining performance
CREATE OR REPLACE FUNCTION i_am_admin() RETURNS boolean LANGUAGE sql STABLE AS $$
  SELECT
    EXISTS (
      SELECT 1 FROM organization_members
      WHERE user_id = auth.uid() AND role = 'admin'
    )
    OR (current_setting('request.jwt.claims', true)::json->>'api_key_id') IS NOT NULL
$$;
```

**How this works:**
- `current_setting('request.jwt.claims', true)` returns NULL (not an error) when the setting doesn't exist — the `true` parameter means "missing_ok"
- For normal user JWTs: `api_company_id` claim is absent → `NULL` → COALESCE falls through to normal path (first branch succeeds)
- For API key JWTs: `auth.uid()` returns the `sub` claim (the API key UUID), which has no match in `organization_members` → first branch returns NULL → COALESCE uses second branch

**`sub` claim constraint:** The JWT's `sub` claim is set to the `api_keys.id` UUID. This UUID is generated by `gen_random_uuid()` and is virtually guaranteed to never collide with a real `auth.users.id`. However, this is a critical invariant: if `sub` ever matched a real user ID, that user's organization_members role would take precedence over the API key claim, potentially returning the wrong company_id. The gateway MUST use the `api_keys.id` (not any user ID) as `sub`.

**Backward compatibility:** The normal user authentication path is checked first via COALESCE and remains identical. The API key path only activates when `api_company_id` is present in the JWT claims — which only the gateway produces.

### View: `api_embeddeds` (hides sensitive fields)

The `embeddeds` table contains sensitive fields (`passkey`, used for XOR encryption) that must not be exposed via the API. A view provides a safe projection.

**Note:** The `embeddeds` table uses `company` (not `company_id`) as the foreign key column name — a legacy naming inconsistency with newer tables.

```sql
CREATE OR REPLACE VIEW api_embeddeds AS
SELECT
  id, company, subdomain, mac_address, status, status_at,
  mdb_diagnostics, created_at, last_restart_reason,
  last_restart_at, online_since
FROM embeddeds;
-- Excludes: passkey (XOR encryption key — NEVER expose)
-- Excludes: owner_id (internal)
```

RLS on views: PostgREST applies the underlying table's RLS policies when querying through a view, so `api_embeddeds` inherits `embeddeds` RLS automatically.

### Rate limit column

```sql
ALTER TABLE api_keys ADD COLUMN IF NOT EXISTS rate_limit integer DEFAULT 100;
-- rate_limit = max requests per minute per key
```

## API Gateway Edge Function

### File: `Docker/supabase/functions/api-v1/index.ts`

Single edge function that handles all `/api/v1/*` requests.

**Request flow:**

1. Handle CORS preflight (`OPTIONS` → `204` with appropriate headers)
2. Extract `X-API-Key` from headers → `401` if missing
3. SHA-256 hash the key, look up in `api_keys` using **service-role client** → `401` if not found or revoked
4. Update `last_used_at` via service-role client (fire-and-forget, non-blocking)
5. Check rate limit counter → `429` if exceeded
6. Parse URL path to determine routing:
   - `/api-v1/docs` → serve Swagger UI
   - `/api-v1/openapi.yaml` → serve OpenAPI spec
   - `/api-v1/actions/*` → proxy to existing edge functions (see Action Routing below)
   - `/api-v1/*` → proxy to PostgREST
7. For CRUD routes, create a short-lived JWT:
   ```json
   {
     "role": "authenticated",
     "sub": "<api_key_id_uuid>",
     "api_company_id": "<company_id>",
     "api_key_id": "<api_key_id_uuid>",
     "iss": "vmflow-api-gateway",
     "iat": <now>,
     "exp": <now + 60>
   }
   ```
8. Forward request to PostgREST with JWT in `Authorization` header + `apikey` header (anon key for Kong)
9. Return PostgREST response to client

**Service-role client:** The gateway uses `SUPABASE_SERVICE_ROLE_KEY` for API key lookups and `last_used_at` updates. This bypasses RLS, which is necessary because the API key validation happens before we know the company context. The service-role client is ONLY used for `api_keys` table operations — all data queries go through PostgREST with the scoped JWT.

**JWT signing:** Uses `JWT_SECRET` environment variable (same secret PostgREST trusts) via the `jose` library to create and sign JWTs.

**Rate limiting:** Simple in-memory Map keyed by `api_key_id` with a sliding window counter (requests in the last 60 seconds). **Limitation:** The Supabase edge runtime may recycle workers between requests (especially in local dev with `per_worker` policy), which resets counters. This is acceptable for v1 — it means rate limiting is best-effort, not strict. A future version could use a Redis counter or DB-based approach.

**Error responses:** Consistent JSON format across all error types:
```json
{
  "error": "rate_limit_exceeded",
  "message": "Rate limit of 100 requests per minute exceeded",
  "retry_after": 23
}
```

Standard error codes: `unauthorized` (401), `forbidden` (403), `not_found` (404), `rate_limit_exceeded` (429), `internal_error` (500).

## API Endpoints

### CRUD Endpoints (via PostgREST proxy)

All CRUD endpoints support PostgREST query syntax: filtering (`?field=eq.value`), ordering (`?order=field.desc`), pagination (`?limit=10&offset=20`), and column selection (`?select=id,name`).

| Method | Endpoint | PostgREST Target | Description |
|--------|----------|------------------|-------------|
| GET/POST/PATCH/DELETE | `/api/v1/machines` | `vendingMachine` | Vending machines |
| GET | `/api/v1/sales` | `sales` | Sales records (read-only — writes via service_role only) |
| GET/POST/PATCH/DELETE | `/api/v1/products` | `products` | Product catalog |
| GET/POST/PATCH/DELETE | `/api/v1/categories` | `product_category` | Product categories |
| GET | `/api/v1/devices` | `api_embeddeds` (view) | Registered devices (read-only, sensitive fields hidden) |
| GET/POST/PATCH/DELETE | `/api/v1/trays` | `machine_trays` | Tray/slot configuration |
| GET/POST/PATCH/DELETE | `/api/v1/warehouses` | `warehouses` | Warehouse locations |
| GET/POST/PATCH/DELETE | `/api/v1/stock-batches` | `warehouse_stock_batches` | Stock batches (FIFO) |
| GET | `/api/v1/paxcounter` | `paxcounter` | Visitor traffic data (read-only — writes via MQTT only) |
| GET | `/api/v1/activity-log` | `activity_log` | Activity/audit log (read-only) |

**Read-only endpoints:** `sales`, `paxcounter`, and `activity-log` are GET-only because the underlying tables only grant `SELECT` to `authenticated` (writes come from MQTT/edge functions, not direct user access). POST/PATCH/DELETE attempts return `403`.

**Path-to-table mapping** is a static allowlist in the gateway. Unknown paths return `404`. This prevents access to tables not explicitly listed (e.g., `api_keys`, `users`, `organization_members`).

**Single resource access:** `GET /api/v1/machines?id=eq.<uuid>` (PostgREST syntax). The gateway also supports convenience syntax `GET /api/v1/machines/<uuid>` which it translates to `?id=eq.<uuid>&limit=1` with a singular JSON response (object instead of array).

### Action Endpoints (via Edge Function proxy)

These endpoints proxy to existing edge functions. The gateway authenticates the API key and then calls the target function internally using the **service-role key** plus the validated `company_id` as context.

| Method | Endpoint | Target Function | Description |
|--------|----------|-----------------|-------------|
| POST | `/api/v1/actions/send-credit` | `send-credit` | Send credit to a device |
| POST | `/api/v1/actions/trigger-ota` | `trigger-ota` | Trigger OTA firmware update |
| POST | `/api/v1/actions/send-config` | `send-device-config` | Send device configuration |

**Action routing implementation:** The gateway does NOT forward the `X-API-Key` header to action functions. Instead, it:
1. Validates the API key and resolves `company_id`
2. Calls the target edge function via internal HTTP with `Authorization: Bearer <SERVICE_ROLE_KEY>`
3. Adds `X-Company-Id: <company_id>` header so the target function knows the company context
4. The target functions (`trigger-ota`, `send-device-config`) need a small modification to accept service-role + company context as an alternative auth path (currently they only accept user JWT)

**Required modifications to existing action functions:**
- `trigger-ota/index.ts` — add: if auth is service-role + `X-Company-Id` header, use that company_id instead of looking up via user JWT
- `send-device-config/index.ts` — same change
- `send-credit/index.ts` — already supports `X-API-Key` natively, but should also accept the service-role path for consistency

### Documentation Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/docs` | Swagger UI (HTML) |
| GET | `/api/v1/openapi.yaml` | OpenAPI 3.x specification |

These endpoints do NOT require an API key (public access).

## Kong Routing

Add to `Docker/volumes/api/kong.yml`:

```yaml
- name: api-v1
  url: http://functions:9000/api-v1
  routes:
    - name: api-v1-route
      strip_path: true
      paths:
        - /api/v1
  plugins:
    - name: cors
      config:
        origins:
          - "*"
        methods:
          - GET
          - POST
          - PATCH
          - DELETE
          - OPTIONS
        headers:
          - Content-Type
          - X-API-Key
        exposed_headers:
          - X-Total-Count
        max_age: 3600
```

**How routing works:**
- Client requests `GET /api/v1/machines?limit=10`
- Kong matches path `/api/v1`, strips it → remaining path is `/machines?limit=10`
- Kong appends to upstream URL `http://functions:9000/api-v1` → final: `http://functions:9000/api-v1/machines?limit=10`
- The edge function router (`main/index.ts`) parses `path_parts[1]` = `api-v1`, loads the `api-v1` function worker
- The `api-v1` function receives the full URL and parses `/api-v1/machines` to extract resource name `machines`

**No Kong auth plugin** — the gateway handles API key validation internally (Kong's key-auth plugin uses a different key store).

## OpenAPI Specification

### File: `Docker/supabase/functions/api-v1/openapi.yaml`

OpenAPI 3.0 specification covering:

1. **Info section** — API name (`VMflow API v1`), version, description, contact
2. **Server URLs** — configurable base URL
3. **Security schemes** — `ApiKeyAuth` (type: `apiKey`, in: `header`, name: `X-API-Key`)
4. **Paths** — All CRUD and action endpoints with:
   - Request/response schemas with examples
   - Query parameter documentation (PostgREST filter syntax guide)
   - Error response schemas (401, 403, 404, 429, 500)
5. **Components/schemas** — Data models: `Machine`, `Sale`, `Product`, `Category`, `Device`, `Tray`, `Warehouse`, `StockBatch`, `PaxCounter`

### Swagger UI

Served at `/api/v1/docs` as an HTML page with **bundled** Swagger UI assets (not CDN — this system is often self-hosted on-premise without internet). The Swagger UI dist files (~3MB) are embedded in the edge function or served from Supabase Storage.

The page includes:
- "Try it out" functionality (users can test with their API key)
- Authentication section: "How to obtain an API key" (link to management UI `/api-keys` page)
- PostgREST query guide: filtering operators (`eq`, `gt`, `gte`, `lt`, `lte`, `like`, `in`), ordering, pagination, column selection
- Rate limiting information
- Error code reference

## Security Considerations

1. **Company isolation** — Enforced by PostgreSQL RLS via `my_company_id()`. The gateway cannot bypass this even if compromised — the JWT it creates is scoped to one company.
2. **Sensitive field protection** — `embeddeds.passkey` and internal fields hidden behind `api_embeddeds` view. Table allowlist prevents access to `api_keys`, `users`, `organization_members`.
3. **Short-lived JWTs** — 60-second expiry. Even if intercepted, the window is minimal.
4. **Rate limiting** — Per-key, configurable via `api_keys.rate_limit` column (best-effort in v1).
5. **Key revocation** — Existing `revoked_at` mechanism. Revoked keys are rejected immediately.
6. **Audit trail** — `last_used_at` updated on every request via service-role client.
7. **No passkey exposure** — The `passkey` field (used for XOR encryption) is never returned by the API.
8. **`sub` claim safety** — The JWT `sub` is set to `api_keys.id` (a `gen_random_uuid()` value), which cannot collide with real `auth.users` IDs. This ensures the `organization_members` lookup in `my_company_id()` returns NULL, falling through to the API key claim path.
9. **Table allowlist** — Only explicitly listed tables are accessible. The gateway rejects requests to unlisted tables with `404`, preventing discovery of internal tables.
10. **Admin-level API keys** — All API keys have full admin CRUD access within their company. There are no read-only or scoped keys in v1. API key holders can create, update, and delete any company resource. This is intentional (keys are created by admins for trusted integrations) but should be communicated clearly in the API documentation.

## Column Naming Note

The database has a legacy naming inconsistency: older tables (`embeddeds`, `vendingMachine`, `products`, `product_category`) use `company` as the FK column, while newer tables (`api_keys`, `warehouses`, `device_provisioning`) use `company_id`. The API does not rename columns — external clients see the column names as they exist in the database. The OpenAPI spec documents the actual column name for each resource.

## Example Usage

```bash
# List all machines
curl -H "X-API-Key: vmf_abc123..." \
  "https://your-server/api/v1/machines"

# Get sales from last 7 days, sorted by date
curl -H "X-API-Key: vmf_abc123..." \
  "https://your-server/api/v1/sales?created_at=gte.2026-04-09&order=created_at.desc&limit=50"

# Create a product
curl -X POST \
  -H "X-API-Key: vmf_abc123..." \
  -H "Content-Type: application/json" \
  "https://your-server/api/v1/products" \
  -d '{"name": "Cola 0.5L", "price": 1.50}'

# Update tray stock
curl -X PATCH \
  -H "X-API-Key: vmf_abc123..." \
  -H "Content-Type: application/json" \
  "https://your-server/api/v1/trays?id=eq.<tray-uuid>" \
  -d '{"current_stock": 15}'

# Send credit to a device
curl -X POST \
  -H "X-API-Key: vmf_abc123..." \
  -H "Content-Type: application/json" \
  "https://your-server/api/v1/actions/send-credit" \
  -d '{"device_id": "...", "amount": 2.00}'

# Get single machine by ID (convenience syntax)
curl -H "X-API-Key: vmf_abc123..." \
  "https://your-server/api/v1/machines/<uuid>"

# Browse API docs (no auth required)
open "https://your-server/api/v1/docs"
```

## Future Enhancements (out of scope for v1)

- **API key scopes** — `read`, `write`, `actions` permissions per key
- **Webhook subscriptions** — Push notifications to external URLs on events (new sale, stock low, etc.)
- **Batch operations** — Bulk create/update endpoints
- **API request logging** — Full request/response audit log table
- **Per-endpoint rate limits** — Different limits for read vs write operations
- **SDK generation** — Auto-generate TypeScript/Python/Go client libraries from OpenAPI spec
- **Strict rate limiting** — Redis or DB-based counter instead of in-memory (survives worker recycling)
