# Per-Device SoftAP Credentials with WiFi QR — Design

**Status:** Proposed
**Date:** 2026-04-30
**Owner:** Lucien

## Problem

All field-deployed VMflow devices currently boot a SoftAP with a hardcoded SSID `VMflow` and password `12345678`, defined in `mdb-slave-esp32s3/main/webui_server.h`. Vending machines are publicly accessible — once that password leaks, every unit in the field is reachable on its captive portal. We need per-device credentials without giving up technician convenience.

## Goals

- Each **claimed** device exposes a unique SoftAP password assigned by the backend.
- **Unclaimed** devices expose an **open** SoftAP (no password). This matches the threat model: in factory / post-factory-reset state the device has no secrets and the captive portal itself is the security boundary; requiring a password adds friction without adding security.
- **Already-claimed devices auto-migrate** to the new model on the first boot after OTA, without needing factory-reset or technician intervention. A one-shot HTTPS sync call brings the per-device password down from the backend, the device reloads its AP config, and from that point on the AP is WPA2-secured.
- Each device's SSID is unique and human-readable (derived from its MAC), regardless of claim state.
- Technician joins the captive portal of a claimed device via a WiFi-QR scan generated in the management UI — no manual typing.
- Password is recoverable via the management UI by an admin once the device is claimed.
- All MQTT payloads stay within the existing 19-byte XOR format (no protocol break).

## Non-Goals

- Production-line sticker printing tooling. The plan documents the serial-log fallback dev workflow; full factory tooling is a separate workstream.
- **(Was deferred, now in scope)** ~~In-field migration of already-claimed devices.~~ Covered by the new `sync-softap-password` endpoint described below; no factory-reset required.
- **Password rotation for already-claimed devices.** Once a device is claimed and has its backend-assigned password, the only way to rotate it in V1 is factory reset → re-claim. A dedicated rotation MQTT command can be added later; the design leaves room for it (next free config-cmd byte is `0x33` since `0x32` is taken by MDB soft reset).
- Column-level RLS for `embeddeds.softap_password`. Frontend gates access behind `isAdmin`; truly hiding the column from viewers (which would require a separate table or column-level grants) is deferred.
- Bluetooth / BLE provisioning path. SoftAP is the only entry point this spec covers.

## Design

### SSID format

```
VMflow-XXXXXX
```

`XXXXXX` = last 3 bytes of `WIFI_IF_STA` MAC, uppercase hex (no separators).

- Example: MAC `f4:12:fa:a1:b2:c3` → `VMflow-A1B2C3`
- 13 chars total (well under the 32-byte WiFi SSID limit).
- Espressif's OUI is identical across all ESP32-S3 → only the last 3 bytes carry discrimination. 24 bits = 16M combinations, no realistic collision risk.
- Uppercase hex avoids `0/O/1/l/I` confusion when read off a printed sticker.
- Computed at every boot via `esp_wifi_get_mac(WIFI_IF_STA, ...)` — no NVS dependency, deterministic across power cycles.

### Password format

12 random characters drawn from the alphabet `ABCDEFGHJKLMNPQRSTUVWXYZabcdefghjkmnpqrstuvwxyz23456789` (24 uppercase + 23 lowercase + 8 digits = 55 chars; ambiguous `0/O/1/l/I` removed). **Generated by the `claim-device` Edge Function** using Web Crypto (`crypto.getRandomValues`) — the same pattern that already produces `passkey` in the same function.

12 chars × ~5.8 bits/char ≈ 70 bits of entropy — enough that brute-forcing is infeasible even with the SoftAP being permanently visible.

The password is **owned by the backend**. It is generated server-side at claim time, persisted in `embeddeds.softap_password`, and returned to the firmware in the claim response body. The firmware stores what it receives in NVS but never generates anything itself. This makes the backend the single source of truth and eliminates the need for any device → backend sync step beyond the existing claim round-trip.

### Auto-migration of already-claimed devices

After OTA, a field device boots with no `softap_pwd` in NVS but a populated `device_id` + `passkey` (i.e. it was claimed long ago). To avoid the open-AP-until-factory-reset trade-off, the firmware spawns a **one-shot sync task** on the first `UPLINK_UP` event that meets these conditions:

- `device_id` is set in NVS (device is claimed).
- `passkey` is set in NVS.
- `srv_url` is set in NVS.
- `softap_pwd` is **absent** or shorter than 8 chars.

The task POSTs to `/functions/v1/sync-softap-password`, gets a backend-generated password, writes it to NVS `softap_pwd`, and re-runs `start_softap()` so the AP transitions from open → WPA2 in place (running `esp_wifi_set_config(WIFI_IF_AP, ...)` on a live AP applies the new auth + deauths existing clients).

**Authentication uses HMAC-SHA256.** The device proves it knows the existing `passkey` (issued by the original `claim-device` call). The signature scheme:

```
signature = base64( HMAC-SHA256( passkey,
                                 device_id + "|" + mac_address + "|" + timestamp ) )
```

The backend re-computes the HMAC using the `passkey` it has stored for `device_id`, compares timing-safe, and accepts if equal AND the timestamp is within ±60 s of server time. The 60 s window matches the firmware's expected SNTP sync latency (the sync task waits up to 30 s for SNTP before sending; in practice it completes in 1–3 s).

**Idempotency.** If the row's `softap_password` is already non-NULL (e.g. previous sync succeeded but the device crashed before writing NVS), the endpoint returns the existing value rather than regenerating — same rule as `claim-device`'s idempotent retry path. A device that loops the sync call always gets the same answer.

**Failure handling.** The task uses the same retry pattern as `provision_claim_task`: 3 attempts with exponential backoff (5 s / 30 s / 120 s). After all attempts fail, the task gives up; the AP stays open until the next boot, at which point the task spawns again. The device remains fully functional (MQTT keeps working independently of the SoftAP password) — the sync only affects post-OTA SoftAP security.

### Open SoftAP for unclaimed devices

Before claim (and on already-claimed devices that haven't been re-claimed since the OTA), the firmware brings up an **open** SoftAP (`WIFI_AUTH_OPEN`, no password). The bootstrap UX is "join the network, captive portal pops up immediately, no password required". This:

- Removes the only piece of public information that needed to be on the sticker (the password). Stickers can be reduced to just the SSID prefix or even pre-printed identically across the entire fleet.
- Eliminates a trivial-but-annoying step from the technician's flow.
- Doesn't reduce real security: `12345678` was a public default everyone in the field knew, so an open AP is functionally equivalent. The captive portal is the actual security boundary — it accepts only well-formed claim codes (validated server-side) and configuration writes that get overridden by the legitimate technician anyway.

### NVS schema additions

In namespace `vmflow`:

| Key | Type | When set | Purpose |
|---|---|---|---|
| `softap_pwd` | string (≤16) | Written by `provision_claim_task` after parsing the claim response | Active SoftAP password. Absence ⇒ open AP. |

Factory reset wipes the entire NVS partition (`nvs_flash_erase()`), so a factory-reset device returns to the open-AP state until the next claim — at which point it receives a fresh backend-generated password.

### Firmware changes

**`webui_server.h`**
- Replace `AP_SSID` macro with helper `softap_get_ssid(char *out, size_t out_len)` (computes from MAC).
- Replace `AP_PASS` macro with helper `softap_get_password(char *out, size_t out_len)` — reads NVS `softap_pwd`; if missing/empty/too-short, writes an empty string into `out` and returns `ESP_OK`. **Never generates random values; never writes NVS; never falls back to a compile-time literal.**

**`webui_server.c::start_softap()`**
- Compute SSID and password via the helpers.
- If `pwd` is empty: configure the AP as `WIFI_AUTH_OPEN`, leave `wifi_config.ap.password` zeroed.
- If `pwd` is non-empty: configure the AP as `WIFI_AUTH_WPA_WPA2_PSK` with the password.
- `ssid_len = strlen(ssid)` either way.

**`webui_server.c::wifi_scan_get_handler` (line 433)**
- The `if (strcmp(ssid, AP_SSID) == 0) continue;` filter compares scan results against the SoftAP's own SSID to hide it from the user-visible list. Replace `AP_SSID` with the dynamic value.

**`mdb-slave-esp32s3.c::provision_claim_task` (response-handling block around lines 2351-2380 for the WiFi path and the parallel cellular block around lines 2494-2530)**
- The request body is **unchanged** — claim still sends only `{short_code, mac_address}`.
- After parsing the response JSON, in addition to the existing `company_id` / `device_id` / `passkey` / `mqtt_host` / `mqtt_port` extraction, also extract the new optional `softap_password` field and write it to NVS `softap_pwd` if present and non-empty.
- The same logic mirrors into both the WiFi and cellular code paths, since the response handler is duplicated between them.

**`mdb-slave-esp32s3.c::softap_sync_task` (NEW)**
- Spawned from the `UPLINK_UP` network event handler (alongside `provision_claim_task`) when the conditions listed in §Auto-migration are met.
- Builds the HMAC-SHA256 signature using `mbedtls_md_hmac()` (mbedTLS is already pulled in via `esp-tls`).
- POSTs `{device_id, mac_address, timestamp, signature}` to `{srv_url}/functions/v1/sync-softap-password` via `esp_http_client` (WiFi) or `modem_https_post_json` (cellular) — same branch logic as `provision_claim_task`.
- On 200 OK with valid `softap_password` in response: writes NVS, calls `start_softap()` to apply the new config, logs `SOFTAP SYNC: applied backend-assigned password`. The currently-connected (if any) SoftAP clients are deauth'd; technicians who happened to join the open AP during the boot window must re-join with the new password (which they can look up in the management UI).
- On 4xx: gives up (probably config drift; manual factory-reset is the escape hatch).
- On 5xx / network error: retries with backoff per §Auto-migration.
- Sets a one-shot in-process flag so it never spawns twice in the same boot.

### Backend changes

**Migration `20260430120000_softap_credentials.sql`**
```sql
ALTER TABLE embeddeds ADD COLUMN IF NOT EXISTS softap_password text;
COMMENT ON COLUMN embeddeds.softap_password IS
  'Per-device SoftAP password reported by firmware on claim or via status payload. NULL for legacy devices that have not yet OTA-ed.';
```

**`claim-device` edge function**
- Add a `generateSoftApPassword()` helper next to the existing `generatePasskey()` (same `crypto.getRandomValues` pattern, different alphabet, length 12). Export so the new sync endpoint can re-use it.
- On the **insert path** (fresh claim): generate a password, write `softap_password` into the new `embeddeds` row, return it in the JSON response next to `passkey`.
- On the **idempotent retry path** (token already used, row already exists): re-read the existing `softap_password` from the row and return it. **Do not regenerate** — that would invalidate the password the device already has stored. For rows that pre-date the column rollout (`softap_password = NULL`), generate-and-store on first retry that hits this path, then return.

**`sync-softap-password` edge function (NEW, `Docker/supabase/functions/sync-softap-password/index.ts`)**

- Accepts POST with body `{device_id, mac_address, timestamp, signature}`.
- Looks up the row by `id = device_id`, fetches `passkey`, `mac_address`, `softap_password`.
- Rejects with 401 if any of: row not found, `mac_address` mismatch, timestamp outside ±60 s of server time, signature doesn't verify against `HMAC-SHA256(passkey, device_id + "|" + mac_address + "|" + timestamp)` (timing-safe compare).
- If `softap_password` is NULL: generate via `generateSoftApPassword()` (imported from `claim-device`), UPDATE the row.
- Always returns `{softap_password}` (idempotent — same value on retries).
- Registered in `Docker/supabase/config.toml` under `[functions.sync-softap-password]` with `verify_jwt = false` (the function does its own auth via HMAC).
- Both **prod** (Docker) and **dev** (Supabase CLI) need the new function entry — see CLAUDE.md "Two Environments" section.

The MQTT webhook surface (`mqtt-webhook`, `_shared/mqtt-publish`, MQTT broker ACL) is still **not** touched — auto-migration runs over HTTPS, not MQTT.

### Backward compatibility

The `claim-device` request body shape is **unchanged** (still `{short_code, mac_address}`), so old firmware still claims successfully. The response gains an optional `softap_password` field that old firmware ignores. The new `sync-softap-password` endpoint is additive — old firmware never calls it.

| Scenario | Behaviour |
|---|---|
| New device, fresh from factory | Open SoftAP (`VMflow-XXXXXX`, no password). Captive portal works as before. |
| New firmware × new backend, fresh claim | `claim-device` generates and returns. Firmware writes NVS. SoftAP is WPA2 from this point on. |
| New firmware × new backend, already-claimed device after OTA | NVS has no `softap_pwd`. SoftAP comes up open. On `UPLINK_UP`, sync task fires, backend generates+returns, firmware writes NVS and reloads AP. SoftAP becomes WPA2 within ~10–20 s of boot. **No factory-reset needed.** |
| New firmware × old backend, fresh claim | Response lacks `softap_password`. NVS stays empty. SoftAP stays open until backend ships. |
| New firmware × old backend, already-claimed device after OTA | Sync task fires, gets 404 (endpoint doesn't exist), retries 3× with backoff, gives up. SoftAP stays open until backend ships. Re-attempted on next boot. |
| Old firmware × new backend, claim | Backend generates+persists, response includes the field, old firmware ignores it. DB has a password the device doesn't know; device's SoftAP is still WPA2-with-`12345678`. Resolved by firmware OTA, which then triggers the sync task to fetch the existing DB value (idempotent — backend returns the same value rather than regenerating). |
| Old firmware × new backend, already-claimed device | Backend has nothing to push (no claim happens, no sync call ever made). Device stays on `12345678`. Resolved by OTA. |
| Old firmware on already-claimed device, no backend changes | Unchanged from today. WPA2 with `12345678`. |
| `embeddeds.softap_password = NULL` (row pre-dates backend rollout) | Generated lazily on the first sync call OR the first idempotent claim retry. Frontend modal shows the open-AP banner until the column is populated. |

The migration is one-way safe: backend can ship before firmware and vice versa, in either order, with no claim-flow breakage. The auto-migration is robust to repeated boots (idempotent on the backend side) and to network failures (retried on next boot).

### Operational notes (out of scope but informed by design)

- **Sticker workflow for new units:** stickers don't need to carry a password at all in V1 — fresh devices have an open SoftAP. The MAC suffix on the SSID is usually known at production time (read via `esptool.py read_mac`) so the SSID can be fully personalised on the sticker. The post-claim per-device password lives only in the management UI; technicians look it up there.
- **In-field migration of already-claimed devices:** automatic. After OTA the device boots, comes online via WiFi or cellular, fires the sync task on `UPLINK_UP`, fetches its backend-generated password, and reloads the SoftAP config in place. There is a brief open-AP window of ~10–20 s between firmware boot and AP reload. Practical security delta during that window is zero (`12345678` was public — the open AP is no worse). After the window the AP is WPA2-secured and stays that way across all subsequent boots.
- **Phone WiFi cache:** technicians who have the legacy `VMflow` SSID saved on their phone will need to "Forget Network" once after the firmware update. The new SSIDs `VMflow-XXXXXX` are distinct entries on the phone, no profile collision once the legacy entry is removed.
- **iOS "unsecured network" warning:** iOS shows a small warning indicator when joining an open WiFi. Technicians joining an unclaimed device's SoftAP will see this — expected and harmless. After claim the warning disappears (WPA2).

## Test Strategy

- **Migration:** apply locally with `supabase migration up`, verify column exists, idempotent re-apply succeeds.
- **`claim-device`:** Deno tests for `generateSoftApPassword` (length, alphabet, no I/O/0/1/l). Integration-style test for the handler shape: a fresh claim returns `softap_password` in response and persists it; an idempotent retry returns the **same** value (not a new one).
- **`sync-softap-password`:** Deno tests for the HMAC verification (valid signature accepted, invalid rejected, timestamp-out-of-window rejected, mac mismatch rejected, unknown device rejected). Plus the idempotency rule (NULL → generated and returned; non-NULL → returned unchanged).
- **Firmware:** Manual on-device QA. SSID-from-MAC + NVS-fallback paths are exercised by "device boots fresh" and "device boots after claim" QA steps. The auto-migration path is exercised by simulating a legacy device (NVS has `device_id` + `passkey` but no `softap_pwd`) and observing the sync task fire on UPLINK_UP.
- **Frontend:** Vitest unit test for the SSID-from-MAC helper (`computeSoftApSsid(mac: string | null): string`). Manual UI walkthrough via `npm run dev`.
- **End-to-end:** factory-reset a development device, confirm SoftAP comes up open, claim it from the captive portal, observe the firmware log show `auth=WPA2`, reboot, confirm the new password matches the management UI. Then **simulate the legacy migration case** by manually clearing NVS `softap_pwd` (via `idf.py monitor` + `nvs_partition_set`), reboot, and confirm the sync task fires, the password is restored from DB, and the AP transitions back to WPA2 within ~20 s.
